# Weather Computer — Protocol Status

This document tracks the implementation status of the Block 1 protocol and related components.

The architecture and protocol decisions referenced here come from the project design discussions.  
Future revisions may change details, but this document reflects the current implementation state.

---

# Block 1 Overview

Block 1 establishes the baseline architecture for the Weather Computer system.

Core design goals:

- Provide situational weather awareness while driving.
- Continue estimating conditions for up to ~2 hours after losing cellular connectivity.
- Use a stored regional weather field and resample it as the vehicle moves.

# Protocol Implementation Status

## Python Reference Implementation

Status: COMPLETE

Location:
weather_protocol/

Capabilities:

- packet encoding and decoding
- strict validation
- CRC verification
- Fixture generation
- Human-readable packet dump tool

Supported packet types:

- RegionalSnapshotV1
- PositionUpdateV1
- AckV1

Binary fixtures are generated for testing and firmware integration.

This remains the protocol truth source.

---

## Rust Firmware Parser

Status: COMPLETE

Capabilities:

- parse the common packet header
- validate magic, version, length, and CRC
- decode packet types:
  - RegionalSnapshotV1
  - PositionUpdateV1
  - AckV1
- strict semantic validation
- tested against Python-generated binary fixtures

The Rust implementation must match the Python reference implementation exactly.

Fixture files generated by Python will be used for Rust parser tests.

---

## Weather Interpolation Engine

Status: COMPLETE

Capabilities:

- spatial interpolation across the 3×3 weather field
- temporal interpolation across 0 / 60 / 120 minute forecast slots
- conservative hazard propagation
- local confidence score computation

Confidence is computed on-device only and is not transmitted in packets.

---

## Rust Device State Layer

Status: COMPLETE

Capabilities:

- stores active RegionalSnapshotV1
- stores latest PositionUpdateV1
- recomputes estimated local conditions
- tracks weather and position timestamps
- exposes current estimated conditions

---

## Persistence Layer

Status: COMPLETE

Capabilities:

- persists the active RegionalSnapshotV1 and latest PositionUpdateV1
- restores state across expected power loss and reboot
- restores the newest valid stored record for weather and position
- falls back to the last good valid record when a newer stored record is corrupted or interrupted
- recomputes estimated local conditions after boot restore
- keeps persistence board-agnostic with an in-memory test backend

Current implementation note:

- hardware-specific flash backend integration is future work

---

## Packet Ingress Layer

Status: COMPLETE

Capabilities:

- accepts raw packet bytes
- routes packets through parser
- applies valid packets to DeviceState
- produces AckV1 responses for accepted or rejected packets

---

## Packet Assembler

Status: COMPLETE

Capabilities:

- accepts arbitrary byte fragments
- skips leading garbage bytes until packet magic is found
- determines packet length from header once available
- buffers fragments until full packet is assembled
- preserves trailing bytes after a completed packet
- emits one complete packet at a time
